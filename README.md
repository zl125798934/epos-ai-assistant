# 保全智能助手（EPOS AI Assistant）

基于 LLM 的保险保全智能助手 Demo，集成了 **RAG 检索增强生成**、**Tool/Function Calling**、**多轮对话记忆管理**、**意图识别与路由** 等核心技术，实现智能问答、自动化客服、业务 Agent 等典型 AI 应用场景。

> 本项目以保险保全真实业务场景为案例，涵盖退保、手机号变更等核心保全业务，展示了 AI 技术在传统金融保险领域的落地实践。

---

## 核心功能

### 1. 智能对话助手
支持自然语言交互，用户可通过对话方式办理保全业务或咨询保全知识。

### 2. 保全知识库检索（RAG）
基于向量检索的保险保全知识问答，能够准确回答保全概念、流程、规则等专业问题，并展示引用来源。

### 3. Agent 保全业务办理
通过 Tool Calling 调用业务工具，实现：
- 保单查询
- 退保申请
- 手机号变更
- 保全进度查询
- 保全取消

### 4. 运营中台可视化
前端左侧展示运营中台，实时反映智能助手对话对实际业务数据的处理结果，形成"对话即操作"的闭环。

## 技术亮点

### 意图识别 + 路由中间层

系统通过 **LLM-based 意图分类器** 自动识别用户意图，将请求路由到对应的专用服务：

```
用户输入
    │
    ▼
┌─────────────────┐
│  意图识别分类器   │  ← LLM 判断用户意图
│  IntentClassifier │
└─────────────────┘
    │
    ├─ KNOWLEDGE ──→ RagAiChatService（知识问答）
    ├─ OPERATION ──→ AgentAiChatService（业务办理）
    └─ CHAT ───────→ ChatOnlyAiChatService（闲聊）
```

- **KNOWLEDGE**：知识问答类，触发 RAG 检索增强回答
- **OPERATION**：操作办理类，触发 Agent 工具调用
- **CHAT**：闲聊类，触发通用对话回复

### 多轮对话记忆管理

基于 LangChain4j 的 `@MemoryId` 注解实现会话级记忆管理，支持跨轮次上下文理解：

```java
Flux<String> chatStream(@MemoryId int memoryId, @UserMessage String userMessage);
```

### Prompt 工程与拆分优化

将原本冗长的单一系统提示词拆分为 **3 个专用 Prompt**，分别绑定到不同服务：

| Prompt 文件 | 用途 | 绑定服务 |
|------------|------|---------|
| `system-prompt-rag.txt` | 知识问答专用 | RagAiChatService |
| `system-prompt-agent.txt` | 业务办理专用 | AgentAiChatService |
| `system-prompt-chat.txt` | 闲聊专用 | ChatOnlyAiChatService |
| `system-prompt.txt` | 精简兜底版本 | 备用 |

### RAG 检索增强生成

- 文档加载：自动加载 `resources/docs/` 目录下的保全知识库文档
- 智能分块：按段落分割（最大 500 字符，重叠 100 字符）
- 向量检索：基于 Embedding 相似度检索，Top-5 结果，阈值 0.75
- 来源引用：回答底部展示引用资料来源

### Tool/Function Calling

通过 LangChain4j 的 `@Tool` 注解定义业务工具，LLM 自动判断调用时机：

| 工具名 | 功能 | 场景 |
|-------|------|------|
| `query_policy_detail` | 查询保单详情 | 用户询问保单信息 |
| `apply_surrender` | 办理退保申请 | 用户申请退保 |
| `apply_phone_change` | 办理手机号变更 | 用户修改手机号 |
| `query_pos_detail` | 查询保全进度 | 用户查询办理进度 |
| `cancel_pos` | 取消保全申请 | 用户取消办理 |

## 业务场景案例

### 场景一：退保流程

```
用户：我要退保
助手：请提供姓名和保单号

用户：张三 80001
助手：确认信息如下：
      保单号：80001
      投保人：张三
      投保日期：2026-04-29
      保单状态：生效中

      确为您的保单。该操作可能影响您的保单权益，是否确认继续？

用户：确认
助手：【调用 apply_surrender】
      退保申请已受理，受理号为#101。
      后续想了解退保的情况，可以输入'退保进度'来查询。
```

**退保业务流程图：**

```
┌──────────┐     ┌──────────┐     ┌──────────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│ 用户发起  │────▶│ 信息收集  │────▶│ 查询保单详情  │────▶│ 结构化展示  │────▶│ 风险提示  │────▶│ 创建保全  │
│ 退保请求  │     │ 姓名/保单号│     │ query_policy  │     │ 保单信息    │     │ 二次确认  │     │ 状态:已受理│
└──────────┘     └──────────┘     └──────────────┘     └──────────────┘     └──────────┘     └────┬─────┘
                                                                                                   │
                                ┌──────────────────────────────────────────────────────────────────┘
                                ▼
                           ┌──────────┐     ┌──────────┐
                           │ 审批处理  │────▶│ 状态变更  │
                           │（人工/自动）│     │ 成功/失败 │
                           └──────────┘     └──────────┘
```

### 场景二：手机号变更流程

```
用户：我要修改手机号
助手：请提供姓名和保单号

用户：张三 80001
助手：确认信息如下：
      保单号：80001
      投保人：张三
      投保日期：2026-04-29
      保单状态：生效中
      当前预留手机号：13800138000

      确为您的保单。是否确认修改手机号？

用户：确认
助手：当前手机号为13800138000，请输入新手机号

用户：13800138001
助手：【调用 apply_phone_change】
      您的申请已受理，请关注受理进度。
```

**手机号变更业务流程图：**

```
┌──────────┐     ┌──────────┐     ┌──────────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│ 用户发起  │────▶│ 信息收集  │────▶│ 查询保单详情  │────▶│ 结构化展示  │────▶│ 确认办理  │────▶│ 收集新手机│
│ 变更请求  │     │ 姓名/保单号│     │ query_policy  │     │ 保单信息    │     │          │     │ 号        │
└──────────┘     └──────────┘     └──────────────┘     └──────────────┘     └──────────┘     └────┬─────┘
                                                                                                   │
                                ┌──────────────────────────────────────────────────────────────────┘
                                ▼
                           ┌──────────┐     ┌──────────┐
                           │ 创建保全  │────▶│ 即时生效  │
                           │ 状态:已受理│     │ 状态:成功 │
                           │          │     │ 同步更新  │
                           └──────────┘     │ 保单数据  │
                                            └──────────┘
                                                 │
                      无需审批，低风险操作，即时生效 ◀┘
```

**与退保流程的关键区别：**

| 特性 | 退保 | 手机号变更 |
|------|------|-----------|
| 风险等级 | 高风险 | 低风险 |
| 二次确认 | **必须** | 无需 |
| 审批环节 | 需要 | **无需** |
| 状态流转 | ACCEPTED → 审批 → SUCCESS | ACCEPTED → **直接 SUCCESS** |
| 数据更新 | 无 | **同步更新保单手机号** |
| 生效时间 | 审批后 | **即时生效** |

## 快速开始

### 环境要求

- JDK 17+
- Maven 3.8+
- Node.js 18+

### 后端启动

```bash
# 1. 克隆项目
git clone <repository-url>
cd epos-ai-assistant

# 2. 配置 API Key（在 application.yml 中）
# langchain4j.community.dashscope.chat-model.api-key: your-api-key

# 3. 编译运行
./mvnw spring-boot:run
```

后端服务将启动在 `http://localhost:8080`

### 前端启动

```bash
cd frontend
npm install
npm run dev
```

前端服务将启动在 `http://localhost:5173`

## 核心设计思想

### 1. 对话即操作
用户通过自然语言对话即可完成复杂的保全业务办理，无需学习复杂的系统操作流程。

### 2. AI 作为智能路由层
意图识别中间层将用户请求智能分发到不同处理链路，实现"一个入口，多种能力"。

### 3. 人机协作
- AI 负责：意图理解、信息收集、流程引导、知识解答
- 人工负责：高风险审批（如退保）、异常处理

### 4. 可扩展的 Tool 体系
通过 `@Tool` 注解可快速扩展新的业务能力，如新增减额交清、保单贷款等保全项目。
